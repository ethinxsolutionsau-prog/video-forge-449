1. Vision & Product Boundaries

What ETHINX IS

A multi-tenant automation operating system for service businesses. Provides lead capture, enrichment, routing, workflow execution, and template-based verticalization—all on a single codebase with strict tenant isolation.

Not a CRM

No contact timeline builder, deal pipelines, or sales forecasting. Integrates with CRMs.

Not a Workflow Builder

No drag-and-drop DAG editor in v1. Workflows are template-defined, config-driven.

Not an Agency

Product-first. Templates are deployable artifacts, not services.

Core Thesis

One platform → many service industries → many templates → one codebase. Tenants get vertical-specific automation in minutes via template installation, without code forks or separate deployments.

Product Boundaries v1

  • Lead Engine: capture, normalize, dedupe, score, assign
  • Workflow Engine: trigger → action, synchronous + queued execution
  • Template System: versioned bundles of workflows, forms, copy, UI config
  • Multi-tenancy with RLS: absolute data isolation
  • Billing via Dodo Payments only
  • Admin, tenant management, audit logs

Data Flow

  1. User authenticates via Supabase Auth → tenant context set
  2. Lead enters via form/API → Edge Function normalizes → RLS inserts
  3. Workflow triggers on lead.created → actions execute
  4. Checkout via Dodo → webhook provisions subscription → RLS grants features
  5. Template install copies config + workflows into tenant scope

3. Multi-Tenancy Model

Tenant Isolation Strategy

Shared database, shared schema, row-level isolation via tenant_id on every tenant-scoped table. Enforced by Postgres RLS. No cross-tenant queries possible from application role.

RLS Approach

  • auth.jwt() → app_metadata.tenant_id set on login
  • All policies check tenant_id = current_tenant_id()
  • Platform Admin bypass via service_role, audited
  • Supabase Storage buckets namespaced by tenant_id

4. Database Schema v1

Core tables with RLS. Extensions: pgcrypto, uuid-ossp.

tenants.sql

CREATE TABLE tenants (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  slug text UNIQUE NOT NULL,
  name text NOT NULL,
  dodo_customer_id text,
  status text NOT NULL DEFAULT 'trialing' CHECK (status IN ('trialing','active','past_due','canceled')),
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);
ALTER TABLE tenants ENABLE ROW LEVEL SECURITY;
CREATE POLICY "tenant_isolation_select" ON tenants FOR SELECT USING (id = current_tenant_id());
CREATE POLICY "tenant_isolation_update" ON tenants FOR UPDATE USING (id = current_tenant_id());

users & memberships.sql

CREATE TABLE memberships (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id uuid NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,

  role text NOT NULL CHECK (role IN ('owner','admin','member','readonly')),
  created_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE(tenant_id, user_id)
);
ALTER TABLE memberships ENABLE ROW LEVEL SECURITY;
CREATE POLICY "members_read_own_tenant" ON memberships FOR SELECT USING (tenant_id = current_tenant_id());
CREATE POLICY "owners_manage_members" ON memberships FOR ALL USING (
  tenant_id = current_tenant_id() AND EXISTS (
    SELECT 1 FROM memberships m WHERE m.tenant_id = current_tenant_id()
    AND m.user_id = auth.uid() AND m.role IN ('owner','admin')

);

Webhook Handlers

// /api/webhooks/dodo
export async function POST(req) {
  const sig = req.headers.get('dodo-signature');
  const raw = await req.text();
  const event = verifyDodoWebhook(raw, sig); // HMAC

  switch(event.type) {
    case 'payment.succeeded':
      await provisionTenant(event.data.customer_id, event.data.subscription_id);
      break;
    case 'subscription.created':
    case 'subscription.updated':
      await upsertSubscription(event.data);
      break;
    case 'subscription.cancelled':
      await markPastDue(event.data.subscription_id);
      break;
  }
  return Response.json({ received: true });
}

8. Workflow Engine Design v1

Simple trigger → action model. No DAG. Runs in Edge Function with queue fallback.

{
  "key": "lead_new_notify",
  "trigger_type": "lead.created",
  "trigger_config": { "min_score": 50 },
  "actions": [
    { "type": "send_email", "config": { "template": "lead_alert" } },
    { "type": "webhook", "config": { "url": "{{tenant.webhook_url}}" } },
    { "type": "assign_user", "config": { "round_robin": true } }
  ]
}

Supported Triggers v1

lead.created, lead.updated, subscription.created, form.submitted

Supported Actions v1

send_email, send_sms, webhook, assign_user, update_lead, create_task

9. Template System

Templates bundle workflows, forms, copy, and UI config for a vertical.

Consult booking, qualification form, proposal send, nurture sequence.

Installation Flow

  1. User selects template → preview manifest
  2. Edge Function clones workflows with tenant_id
  3. Forms and copy written to tenant config
  4. template_installations record created

10. Lead Engine v1 Specification

Inputs

Web forms, Facebook Lead Ads, Email parsing, Phone (Twilio webhook)

Process

Normalize → Deduplicate → Score → Assign

Outputs

CRM record, Notification, Workflow trigger

API Endpoints

POST /api/leads/ingest  # public, HMAC signed
GET  /api/leads
PATCH /api/leads/:id
POST /api/leads/:id/events

Acceptance Criteria

  • Dedupe within 5 min window by email+phone hash
  • Score 0-100 based on source + fields completeness
  • Assignment respects user capacity and round-robin
  • All writes audit logged
